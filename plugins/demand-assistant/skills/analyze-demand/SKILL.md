# 需求分析知识库

## 需求文档标准结构

需求文档采用以下标准结构，分为三大板块：

### 一、需求背景（必填）

| 字段 | 必填 | 说明 |
|------|------|------|
| 需求来源 | 是 | 提出需求的部门/团队 |
| 适用部门 | 是 | 需求上线后使用的部门 |
| 调研人群 | 否 | 参与需求调研的人员 |
| 业务背景 | 是 | 当前现状、存在的问题和痛点 |
| 实现目标 | 是 | 本次需求要达成的目标和预期效果 |

### 二、详细描述（必填）

每个功能点包含：
- **功能名称**：简明扼要的功能描述
- **所在路径**（选填）：系统菜单或页面路径
- **实现步骤**：编号列表，包含：
  - 规则描述
  - 触发节点和条件
  - 计算规则（公式）
  - 约束条件
  - 数据来源（SQL、接口等）

### 三、补充说明（选填）

- 影响范围
- 风险点
- 上下游依赖
- 需要确认的事项

## 需求边界澄清方法论

### 六大澄清维度

1. **需求来源与干系人**
   - 提出方：哪个部门/团队
   - 使用角色：运营、仓库、财务、开发等
   - 影响范围：是否跨部门

2. **业务背景与痛点**
   - 现状：当前流程是怎样的
   - 痛点：存在什么问题
   - 频率：问题发生的频率和影响面

3. **功能范围与边界**
   - 包含：明确列出要做的功能点
   - 不包含：明确排除的内容
   - 联动：是否涉及其他系统/模块

4. **业务规则与约束**
   - 计算公式：数值计算逻辑
   - 阈值：判断条件和临界值
   - 异常处理：兜底和容错逻辑
   - 权限：角色权限要求

5. **数据依赖**
   - 数据源：依赖的表、接口、系统
   - 时效性：数据更新频率和延迟要求
   - 数据量：预估数据规模

6. **验收标准**
   - 功能验收：核心验证点
   - 非功能要求：性能、兼容性、安全性
   - 回归影响：是否影响现有功能

### 澄清问题分级

- **P0 必须确认**：缺失会导致需求文档无法生成或严重偏离
- **P1 建议确认**：有部分信息但存在歧义，影响文档准确性
- **P2 可选确认**：细节优化类问题，可后续迭代

## 标注规范

| 标注 | 含义 | 使用场景 |
|------|------|----------|
| `[待补充]` | 原始材料中完全未提及 | 缺少对应信息时 |
| `[推断]` | 根据上下文合理推测 | 有间接信息可推断时 |
| `[需确认]` | 存在矛盾或不确定 | 信息冲突或模糊时 |
